# Implementation Plan: Stripe & Braintree Payment APIs with Merchant/Branch Domain

**Branch**: `007-stripe-braintree-payment-apis` | **Date**: 2026-04-20 | **Spec**: [spec.md](spec.md)

## Summary

Extend the existing Stripe and Braintree payment modules to support per-merchant and per-branch credential resolution, introduce the `MerchantBranch` entity as a sub-entity of `MerchantApplication`, add a `MerchantCredentialResolver` cross-module interface, and update `MerchantProduct` to carry a `merchantBranchId` FK. All three payment endpoints (`stripe/create-intent`, `braintree/client-token`, `braintree/checkout`) are updated to accept `merchantId`, optional `branchId`, and optional `productId`.

## Implementation Phases

### Phase A — Merchant Domain Extension

**A1. `MerchantBranch` entity**

New JPA entity `MerchantBranch` in `merchant.domain`:
- Table: `merchant_branches` (schema `blitzpay`)
- Fields: `id` (UUID PK), `merchantApplicationId` (UUID FK), `name`, `active`, nullable Stripe fields (`stripeSecretKey`, `stripePublishableKey`), nullable Braintree fields (`braintreeMerchantId`, `braintreePublicKey`, `braintreePrivateKey`, `braintreeEnvironment`), `createdAt`, `updatedAt`
- Indexes: `idx_merchant_branches_merchant_application_id`

**A2. `MerchantApplication` credential columns**

Add nullable columns via new `@Column` fields on existing entity:
`stripeSecretKey`, `stripePublishableKey`, `braintreeMerchantId`, `braintreePublicKey`, `braintreePrivateKey`, `braintreeEnvironment`

**A3. `MerchantProduct.merchantBranchId`**

Add nullable `@Column(name = "merchant_branch_id") var merchantBranchId: UUID? = null` to `MerchantProduct`.

**A4. `MerchantBranchRepository`**

```kotlin
interface MerchantBranchRepository : JpaRepository<MerchantBranch, UUID> {
    fun findAllByMerchantApplicationIdAndActiveTrue(merchantApplicationId: UUID): List<MerchantBranch>
    fun findByIdAndActiveTrue(id: UUID): MerchantBranch?
}
```

**A5. `MerchantCredentialResolver` @NamedInterface**

In `merchant.api`:
```kotlin
data class StripeCredentials(val secretKey: String, val publishableKey: String)
data class BraintreeCredentials(
    val merchantId: String, val publicKey: String, val privateKey: String, val environment: String
)

interface MerchantCredentialResolver {
    fun resolveStripe(merchantId: UUID, branchId: UUID?): StripeCredentials?
    fun resolveBraintree(merchantId: UUID, branchId: UUID?): BraintreeCredentials?
    fun resolveBranch(merchantId: UUID, branchId: UUID?, productId: UUID?): UUID?
}
```

Register in `merchant.api.package-info.kt` via `@NamedInterface`.

**A6. `MerchantCredentialResolverImpl`**

Resolution logic:
1. If `branchId` is provided, look up `MerchantBranch`; use its credentials if non-null.
2. If branch credentials are absent or `branchId` is null, fall back to `MerchantApplication` credential fields.
3. If merchant fields are also absent, return `null` (caller returns 503).

Branch resolution:
1. If `branchId` not null → return `branchId` (validate it belongs to the merchant).
2. If `productId` not null → load `MerchantProduct`; return `product.merchantBranchId`.
3. Otherwise → return `null` (controller returns 400).

**A7. `MerchantBranchService` + `MerchantBranchController`**

Service: create, list, get by ID (all scoped to `merchantApplicationId`).
Controller: `POST /v1/merchants/{merchantId}/branches` (201), `GET /v1/merchants/{merchantId}/branches` (200).
Response shape: `BranchResponse` — never returns secret keys or private keys; uses `hasStripeCredentials: Boolean` and `hasBraintreeCredentials: Boolean` flags instead.

### Phase B — Stripe Module Update

**Goal**: Each PaymentIntent carries `merchantId`, `branchId`, and `productId` in Stripe's `metadata` field — visible in the Stripe dashboard and in all webhook payloads. Credentials are resolved per-request via `RequestOptions`; the global key assignment is removed.

**B1. Remove global key assignment**

In `StripeConfig.kt`, remove `@PostConstruct fun configureStripe()`. The global `Stripe.apiKey` is no longer set. All calls use per-request `RequestOptions`.

**B2. Update `StripePaymentService`**

New signature:
```kotlin
fun createIntent(
    amount: Double,
    currency: String,
    credentials: StripeCredentials,
    merchantId: UUID,
    branchId: UUID?,
    productId: UUID?,
): Mono<StripeIntentResult>
```

Build Stripe metadata and `RequestOptions` inside the `Mono.fromCallable` block:
```kotlin
val requestOptions = RequestOptions.builder()
    .setApiKey(credentials.secretKey)
    .build()

val params = PaymentIntentCreateParams.builder()
    .setAmount(amountInSmallestUnit)
    .setCurrency(currency.lowercase())
    .setAutomaticPaymentMethods(
        PaymentIntentCreateParams.AutomaticPaymentMethods.builder().setEnabled(true).build()
    )
    .putMetadata("merchantId", merchantId.toString())
    .apply { branchId?.let { putMetadata("branchId", it.toString()) } }
    .apply { productId?.let { putMetadata("productId", it.toString()) } }
    .build()

val intent = PaymentIntent.create(params, requestOptions)
log.info("stripe create_intent id={} amount={} currency={} merchantId={} branchId={} productId={}",
    intent.id, amount, currency.lowercase(), merchantId, branchId, productId)
```

Return `StripeIntentResult(clientSecret, credentials.publishableKey)`.

**B3. Update `StripePaymentController`**

New request shape:
```kotlin
data class CreateIntentRequest(
    val amount: Double? = null,
    val currency: String? = null,
    val merchantId: UUID? = null,
    val branchId: UUID? = null,
    val productId: UUID? = null,
)
```

Controller logic:
1. Validate `merchantId` present → 400 if missing.
2. Resolve `branchId` via `MerchantCredentialResolver.resolveBranch(merchantId, branchId, productId)` → 400 if null.
3. Load product if `productId` present; derive/validate `amount`; log WARN if explicit amount differs from `product.unitPrice`.
4. Resolve `StripeCredentials` via `MerchantCredentialResolver.resolveStripe(merchantId, resolvedBranchId)` → 503 if null.
5. Call `stripePaymentService.createIntent(amount, currency, credentials, merchantId, resolvedBranchId, productId)`.

### Phase C — Braintree Module Update

**Goal**: Each Braintree transaction carries `merchantId`, `branchId`, and `productId` as the `orderId` and custom fields — surfaced in the Braintree Control Panel and in webhook events. Credentials are resolved per-request via a cached `BraintreeGatewayFactory`. The three payment modules remain fully independent.

**C1. `BraintreeGatewayFactory`**

```kotlin
@Component
class BraintreeGatewayFactory {
    private val cache = ConcurrentHashMap<String, BraintreeGateway>()

    fun get(credentials: BraintreeCredentials): BraintreeGateway =
        cache.computeIfAbsent(credentials.fingerprint()) { _ ->
            BraintreeGateway(
                Environment.parseEnvironment(credentials.environment.uppercase()),
                credentials.merchantId, credentials.publicKey, credentials.privateKey
            )
        }

    private fun BraintreeCredentials.fingerprint() = "${merchantId}:${publicKey}:${environment}"
}
```

**C2. Update `BraintreePaymentService`**

Remove `Optional<BraintreeGateway>` constructor injection; inject `BraintreeGatewayFactory` instead.

```kotlin
fun isConfigured(credentials: BraintreeCredentials?): Boolean = credentials != null

fun generateClientToken(credentials: BraintreeCredentials): Mono<String>

fun checkout(
    nonce: String,
    amount: Double,
    currency: String,
    credentials: BraintreeCredentials,
    merchantId: UUID,
    branchId: UUID?,
    productId: UUID?,
    invoiceId: String? = null,
): Mono<BraintreeCheckoutResult>
```

Build the `TransactionRequest` with traceability fields:
```kotlin
// orderId encodes the primary external reference (product if present, else invoiceId)
val externalRef = productId?.toString() ?: invoiceId ?: ""

val request = TransactionRequest()
    .amount(BigDecimal(formattedAmount))
    .paymentMethodNonce(nonce)
    .orderId(externalRef)          // Braintree's native external reference field
    // customFields require pre-registration in Braintree Control Panel;
    // if registered, add: .customField("merchant_id", merchantId.toString())
    //                       .customField("branch_id", branchId?.toString() ?: "")
    .options().submitForSettlement(true).done()
```

Log on success:
```kotlin
log.info("braintree checkout OK tx={} amount={} currency={} merchantId={} branchId={} productId={} invoice={}",
    tx.id, formattedAmount, currency, merchantId, branchId, productId, invoiceId ?: "n/a")
```

`BraintreeCheckoutResult.Success` gains `merchantId: UUID` and `branchId: UUID?` fields.

**C3. Update `BraintreeConfig`**

Remove the `@Bean @ConditionalOnProperty` `BraintreeGateway` bean — gateway creation is now the factory's responsibility.

**C4. Update `BraintreePaymentController`**

New request shapes:
```kotlin
data class ClientTokenRequest(
    val merchantId: UUID? = null,
    val branchId: UUID? = null,
)

data class CheckoutRequest(
    val nonce: String? = null,
    val amount: Double? = null,
    val currency: String? = null,
    val merchantId: UUID? = null,
    val branchId: UUID? = null,
    val productId: UUID? = null,
    val invoiceId: String? = null,
)
```

Controller follows the same branch/credential resolution pattern as `StripePaymentController` (steps 1–5 above). `CheckoutSuccessResponse` gains `merchantId: UUID` and `branchId: UUID?` fields.

### Phase E — TrueLayer Module Update

**Goal**: TrueLayer payments already use `.metadata()` on `CreatePaymentRequest`. Extend `PaymentRequested` with optional `merchantId`, `branchId`, and `productId` so the publisher (qrpay module) can supply them and `PaymentService` forwards them as TrueLayer metadata — keeping the module self-contained and independent.

**E1. Extend `PaymentRequested`**

In `payments.truelayer.api`:
```kotlin
data class PaymentRequested(
    var paymentRequestId: UUID?,
    val orderId: String,
    val amountMinorUnits: Long,
    val currency: String,
    val userDisplayName: String,
    val redirectReturnUri: String,
    val merchantId: UUID? = null,    // new — optional for backward compat
    val branchId: UUID? = null,      // new
    val productId: UUID? = null,     // new
)
```

**E2. Update `PaymentService.startPayment`**

Extend the existing `.metadata()` map to include the new fields:
```kotlin
.metadata(buildMap {
    put("paymentRequestId", paymentRequest.paymentRequestId.toString())
    put("orderId", paymentRequest.orderId)
    paymentRequest.merchantId?.let { put("merchantId", it.toString()) }
    paymentRequest.branchId?.let  { put("branchId",  it.toString()) }
    paymentRequest.productId?.let { put("productId", it.toString()) }
})
```

Update the log line to include the new fields:
```kotlin
log.info(
    "truelayer createPayment request paymentRequestId={} orderId={} amountMinor={} currency={} merchantAccount={} merchantId={} branchId={} productId={}",
    ..., paymentRequest.merchantId, paymentRequest.branchId, paymentRequest.productId,
)
```

No other TrueLayer files change. The `qrpay` module that publishes `PaymentRequested` can populate the new optional fields when it has a product/merchant context; existing callers that omit them continue to work.

## Key Design Decisions Summary

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Merchant entity | Extend `MerchantApplication` | Mirror confirmed; avoids duplicate entity and sync overhead |
| Stripe credential scoping | `RequestOptions` per call | Thread-safe; SDK-native; avoids JVM-global key race |
| Stripe transaction traceability | `PaymentIntentCreateParams.putMetadata()` | Native Stripe metadata field; appears in dashboard and all webhooks |
| Braintree gateway reuse | Factory + `ConcurrentHashMap` cache | Avoids connection pool exhaustion per request |
| Braintree transaction traceability | `TransactionRequest.orderId()` + custom fields | `orderId` is Braintree's native external reference; custom fields require Control Panel pre-registration |
| TrueLayer transaction traceability | Extend `.metadata()` map on `CreatePaymentRequest` | TrueLayer already uses metadata; adding fields is additive and backward compatible |
| Payment module independence | Each module consumes `merchant.api` only; none reference each other | Spring Modulith boundary enforcement; `ApplicationModules.verify()` gates this |
| Cross-module interface | `@NamedInterface` in `merchant.api` | Required by Spring Modulith; matches existing project pattern (`payments.truelayer`) |
| `MerchantProduct` migration | Nullable `merchantBranchId` | `ddl-auto: update` safe; backward compatible with existing rows |
| Credential secrets in responses | Never returned; boolean flags only | FR-008, SC-005 |
